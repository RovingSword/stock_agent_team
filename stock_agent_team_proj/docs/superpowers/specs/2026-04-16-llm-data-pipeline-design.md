# LLM 分析链路数据修复设计

## 背景
当前 `analyze_llm` 链路中，技术分析员、情报员、风控官、基本面分析员并没有直接拿到 `DataFetcher` 返回的真实原始数据，而是拿到了规则引擎汇总后的评分、评论和要点。与此同时，部分字段来源错误或根本不存在，导致四个角色共同出现“拿不到正确数据”的问题。

该问题由三层原因叠加：

1. `web/api/analyze.py` 在构建 LLM 上下文时错误复用了规则引擎摘要数据。
2. `score_breakdown` 本身只包含分数和权重，不包含 LLM 期望的原始分析字段。
3. 缺数据时当前逻辑倾向于用默认值或伪造完整上下文，掩盖真实数据缺失。

## 目标

本次修复的目标是：

1. 让四个 LLM 分析角色拿到按职责拆分的真实原始数据。
2. 让规则引擎结果只作为参考结论，而不是冒充原始数据。
3. 缺失数据时显式标记，避免把伪造默认值继续传递给 LLM。
4. 用最小范围修改修复线上主要问题，并建立回归测试。

本次修复不包含：

1. 全面重写规则引擎四个规则型 Agent 的内部分析逻辑。
2. 一次性消除所有历史硬编码字段。
3. 增加新的第三方数据源。

## 方案选项

### 方案 A：只修提示词，不改数据装配

做法：
保留当前 `analysis_data` 结构，只在 prompt 中提醒 LLM “参考信息可能不完整”。

优点：
改动最小。

缺点：
根因不变，LLM 仍然拿不到真实数据；只能缓解，不能修复。

### 方案 B：改造 `analyze_llm` 数据装配层，按角色注入真实数据

做法：
在 `web/api/analyze.py` 中显式调用 `DataFetcher.get_full_data()`，构造面向四个角色的原始上下文字段；规则引擎结果仅作为 supplemental reference 一并传入。

优点：
直接修复根因；改动集中；风险可控；便于测试。

缺点：
需要同步调整 prompt 构造和测试用例。

### 方案 C：先重构规则引擎四个 Agent，再统一复用其结果

做法：
先清理规则引擎内部所有默认值和硬编码，再把规则引擎结果作为 LLM 唯一输入。

优点：
体系更统一。

缺点：
范围过大，修复周期长，容易引入新的行为变化。

## 推荐方案

选择方案 B。

原因：
用户当前问题是“四个分析角色都拿不到正确数据”，共享根因在 LLM 数据装配层。优先修复这条公共链路，能最快恢复主要能力；同时保留规则引擎作为参考信息，减少行为突变。

## 设计

### 1. 数据装配边界

新增一个集中式上下文构造步骤，位置放在 `web/api/analyze.py` 中，职责如下：

1. 调用 `data_fetcher.get_full_data(stock_code)` 获取完整数据。
2. 提取当前价，优先顺序为：
   - `quote.current_price`
   - `technical.current_price`
   - 若均不存在则为 `None`
3. 为每个角色构造单独的 `role_payload`：
   - `technical`: 技术指标、K 线摘要、实时价格
   - `intelligence`: 资金流、北向资金、新闻、市场数据
   - `risk`: 市场数据、实时价格、基础风险提示、可用的基本面摘要
   - `fundamental`: 财务数据、估值数据、实时价格
4. 同时附带 `rule_reference`，即规则引擎给出的分数、要点、风险点和执行建议。

### 2. Prompt 输入结构

当前 `_build_llm_agent_prompt()` 只接受单一 `role_data`，后续改为接收两类信息：

1. `raw_data`: 真实原始数据
2. `rule_reference`: 规则引擎参考结论

Prompt 语义改为：

1. 先分析真实原始数据。
2. 仅将规则引擎输出作为参考，不得把参考结论当作原始事实。
3. 若某些关键字段缺失，应在结论中明确说明“数据不足”。

### 3. 缺失数据策略

本次不再向 LLM 伪造“完整数据”。统一策略如下：

1. 原始字段缺失时保留 `None`、空字典或空列表。
2. 在 prompt 中明确指出“以下字段为空，表示当前未获取到真实数据”。
3. 不再从 `score_breakdown` 派生伪 `comment/key_points` 去替代原始数据。

这样可避免 LLM 基于错误事实产出高置信度分析。

### 4. 当前价与执行数据修正

`generate_sse_events()` 当前从 `rule_decision.execution.get("current_price", 0)` 读取现价，但规则引擎执行结果并不保证该字段存在。修复方式：

1. 优先从 `quote.current_price` 读取。
2. 次选 `technical.current_price`。
3. 最后才回退到规则引擎执行字段。

## 测试策略

采用测试优先的最小回归覆盖：

1. 新增或更新 `web/api/analyze.py` 相关测试。
2. 首先写失败测试，验证：
   - LLM 上下文使用了 `DataFetcher` 真实数据而不是仅规则引擎评分。
   - `current_price` 优先来自真实行情。
   - 原始数据缺失时，传入的是空值/缺失标记而不是伪造完整内容。
3. 修复实现后，保证原有 SSE 决策测试继续通过。

## 风险与兼容性

主要风险：

1. prompt 变长，可能略微增加 token 消耗。
2. 某些依赖旧 prompt 格式的测试需要同步调整。
3. 如果运行环境本身缺少 `numpy/requests/akshare/efinance`，真实数据仍会为空，但这会被显式暴露出来，而不是被错误隐藏。

兼容性策略：

1. 保留 `rule_reference`，避免模型完全失去参考。
2. 不改动 SSE 返回结构，前端消费接口保持不变。

## 实施步骤

1. 先补失败测试，覆盖真实数据装配与缺失数据场景。
2. 修改 `web/api/analyze.py`，引入完整数据抓取与角色数据映射。
3. 调整 prompt 构造函数，分离 `raw_data` 与 `rule_reference`。
4. 运行相关测试并检查回归。
5. 如有必要，再补一条日志，标记当前分析是否存在关键数据缺失。

## 验收标准

满足以下条件视为修复完成：

1. 四个 LLM 角色的 prompt 中都包含与各自职责对应的真实原始数据。
2. `current_price` 不再默认变成 `0`。
3. 缺失数据不会被伪造为完整分析事实。
4. 相关自动化测试通过。
